# apps/web/src/app/drops

Drop list + drop detail pages, plus the purchase route handler.

## Files

| Path                                     | Runtime          | Purpose                                                    |
|------------------------------------------|------------------|------------------------------------------------------------|
| `page.tsx`                               | server           | `/drops` — lists all drops from the DB                     |
| `[id]/page.tsx`                          | server           | `/drops/[id]` — single drop detail                         |
| (the `/api/drops/*` handlers live at `src/app/api/drops/*`) | |

## Server vs client split

- The page file (`page.tsx`) is a server component. It runs on the request path, reads the current user, queries Prisma for drops, and hands the data to a client component for render.
- The client component (`DropsList`, `DropDetail`) handles polling or WebSocket subscription for live updates. They never access Prisma or the JWT directly.

## Purchase transaction flow

`POST /api/drops/:id/purchase` — the concurrency-critical endpoint. One Prisma interactive transaction, `ReadCommitted` isolation, 10s timeout:

1. Row-lock the drop: `SELECT ... FROM drops WHERE id = :id FOR UPDATE`. Concurrent callers serialise here.
2. Guards (all inside the lock):
   - `now >= startsAt AND now < endsAt` — drop is live now (server is authoritative; client clock skew is irrelevant).
   - `remaining > 0` — not sold out.
   - `count(UserPack WHERE userId=:u AND dropId=:d) < 5` — per-user cap (D3).
   - `User.balance >= tier_price` — enough funds.
3. Updates:
   - `drops.remaining -= 1`, `status = SOLD_OUT` when it hits 0.
   - `users.balance -= price`.
   - Insert `UserPack`.
   - `pickCards(tier, cardPool)` → 5 cards; insert 5 `pack_cards` with `pricedCaptured` = each card's `basePrice` at the moment of sale.
   - Insert `Ledger` row: `delta = -price`, `reason = PACK_PURCHASE`, `refType = 'UserPack'`, `refId = userPack.id`, `balanceAfter = new balance`.
4. Commit.
5. After commit only: fire-and-forget `emitToRoom('drop:<id>', 'inventory_update', {remaining})`. WS failure is logged but never rolls back the DB.

## Error codes

| Code                 | HTTP | When                                                  |
|----------------------|------|--------------------------------------------------------|
| `unauthorized`       | 401  | No valid JWT cookie                                    |
| `invalid_id`         | 400  | `:id` isn't a UUID                                     |
| `not_found`          | 404  | Drop row doesn't exist, or user row disappeared        |
| `not_live`           | 409  | `now < startsAt` or `now >= endsAt`                    |
| `sold_out`           | 409  | `remaining <= 0` at the moment of lock                 |
| `over_limit`         | 402  | User already owns 5 packs from this drop               |
| `insufficient_funds` | 402  | Balance < tier price                                   |
| `internal_error`     | 500  | Unexpected. Transaction rolls back.                    |

All failures inside the transaction leave the DB unchanged — either the whole thing commits (inventory decremented + ledger entry + cards) or nothing does.

## Rarity weights source of truth

`src/lib/rarity-weights.ts` — auto-generated by `pnpm calibrate:rarity`. Do not hand-edit.

## User flow for this folder

1. User logs in → lands on `/`.
2. Clicks **Drops** in the header → `/drops` renders the grid of drops with countdowns and "Live"/"Sold out"/"Scheduled" badges. Client polls `/api/drops` every 30s to catch status transitions.
3. Clicks a drop → `/drops/[id]` loads with the initial `remaining`. Client connects to the WS service and joins `drop:<id>`, so other buyers' purchases update the counter live.
4. Clicks **Buy 1 Pack** → `<ConfirmPurchaseModal>` shows price, balance, and published odds for this tier.
5. Clicks **Buy 1 Pack** in the modal → POST `/api/drops/:id/purchase` → on 200 OK the browser is pushed to `/me/packs`; on any 4xx the modal shows a human-readable error.
